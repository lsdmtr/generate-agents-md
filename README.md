# generate-agents-md

`generate-agents-md` is a Codex skill for creating or revising durable, repository-level agent instruction files such as `AGENTS.md`, `CLAUDE.md`, and `GEMINI.md`.

`generate-agents-md` 是一个 Codex skill，用于创建或修订长期有效的仓库级 Agent 开发规范文件，例如 `AGENTS.md`、`CLAUDE.md`、`GEMINI.md`。

It is designed for real repositories across different technology stacks. Instead of copying a fixed template, it guides Codex to inspect the repository first, identify the project type, choose the right strictness profile, and then generate actionable project rules.

它面向不同技术栈的真实项目，不是简单复制固定模板。它会指导 Codex 先理解仓库事实，再识别项目类型、选择合适的规范强度，最后生成可执行、可长期维护的项目级开发规范。

## What This Skill Solves / 解决什么问题

Many repositories need an `AGENTS.md` file, but a useful one cannot be generated by filling in a generic checklist. A good project instruction file should tell future agents where requirements live, which files can be edited, where new code belongs, what documentation must be updated, which commands prove the work, and what should never be treated as project-wide policy.

很多仓库都需要 `AGENTS.md`，但真正有用的规范不能靠通用清单填空生成。好的项目级规范应该告诉后续 Agent：需求在哪里、哪些文件可以改、代码应该放到哪里、哪些文档必须同步、用什么命令证明改动有效，以及哪些临时需求不能被写成项目常态。

This skill helps Codex produce that kind of file by forcing a repository-first process:

这个 skill 会强制 Codex 采用“先理解仓库，再生成规范”的流程：

- Inspect existing instructions, manifests, source layout, tests, docs, requirements, and verification commands.
- Read and preserve durable project-specific rules instead of replacing them with generic prose.
- Choose a strictness profile that matches the repository type.
- Generate concrete good and bad examples for fragile conventions.
- Validate the result with a bundled smoke-test script.

- 先检查已有规范、工程配置、源码结构、测试、文档、需求目录和验证命令。
- 保留长期有效的项目规则，而不是用通用文字覆盖项目事实。
- 根据仓库类型选择合适的规范强度。
- 对容易误解的规则提供合格示例和不合格示例。
- 使用内置脚本对生成结果做结构化烟测。

## Key Features / 核心能力

- **Technology-stack aware**: Supports frontend, backend, SDK, mobile, CLI, data, infrastructure, docs, and mixed repositories.
- **Profile based**: Uses `full`, `sdk`, `app`, `backend`, `cli`, `data`, `infra`, or `light` to avoid overfitting one project style to every repo.
- **Bilingual first prompt**: Asks the user whether to use Chinese or English before other clarifying questions when the language is not obvious.
- **Project-wide by default**: Prevents one ticket, version, sprint, branch, or temporary plan from becoming a permanent project rule.
- **Examples required**: Encourages concrete good and bad examples for docs, API contracts, comments, import/export rules, verification, CLI behavior, infra safety, or data reproducibility.
- **Validation included**: Bundles a zero-dependency Python validator for structural checks and task-specific forbidden wording.

- **技术栈感知**：支持前端、后端、SDK、移动端、CLI、数据、基础设施、文档和混合型仓库。
- **按档位生成**：通过 `full`、`sdk`、`app`、`backend`、`cli`、`data`、`infra`、`light` 避免把某一种项目规范强行套到所有仓库。
- **优先确认语言**：如果语言不明确，会先用中英双语询问用户希望使用中文还是英文。
- **默认项目级长期规范**：避免把某个 ticket、版本、迭代、分支或临时方案写成项目常态。
- **要求示例**：鼓励对文档、接口合同、注释、import/export、验证命令、CLI 行为、基础设施安全、数据可复现等高风险规则提供合格和不合格示例。
- **内置校验**：提供零依赖 Python 校验脚本，用于检查结构完整性和禁止出现的一次性需求措辞。

## How It Works / 工作方式

The skill guides Codex through this order:

该 skill 会让 Codex 按以下顺序工作：

1. Ask which user-facing language to use if the user has not already specified it.
2. Inspect existing `AGENTS.md`, `CLAUDE.md`, `GEMINI.md`, project docs, requirements, source layout, tests, scripts, manifests, CI, and verification commands.
3. Identify repository facts: stack, runtime, package manager, framework, formatter, linter, test runner, build path, deployment path, boundaries, and risk areas.
4. Classify the repository into a profile.
5. Ask only for non-discoverable intent, such as whether strict docs-before-code or container-only verification is desired.
6. Generate or revise the instruction file using the bundled blueprint.
7. Validate the generated file with the bundled script and repo-appropriate checks.

1. 如果用户没有指定语言，先询问后续提示和生成文档使用中文还是英文。
2. 检查已有 `AGENTS.md`、`CLAUDE.md`、`GEMINI.md`、项目文档、需求目录、源码结构、测试、脚本、工程配置、CI 和验证命令。
3. 识别仓库事实：技术栈、运行时、包管理器、框架、格式化工具、lint、测试框架、构建路径、部署路径、工作边界和风险点。
4. 将仓库归类到合适 profile。
5. 只询问无法从仓库推导出的意图，例如是否要求严格文档先行、TDD 或只允许容器内验证。
6. 使用内置蓝图生成或修订规范文件。
7. 使用内置脚本和仓库自身检查命令校验结果。

## Profiles / 规范强度档位

| Profile | English | 中文 |
| --- | --- | --- |
| `full` | Mature product, platform, SDK, or service with durable docs and public contracts. | 有长期文档和公开合同的成熟产品、平台、SDK 或服务仓库。 |
| `sdk` | Library, package, SDK, plugin, or reusable module where exports and adapters matter. | 库、包、SDK、插件或可复用模块，重点关注导出面、适配层和使用示例。 |
| `app` | Frontend, mobile, desktop, or full-stack application where flows and UI states matter. | 前端、移动端、桌面端或全栈应用，重点关注用户流程、页面状态和平台行为。 |
| `backend` | API, worker, job, service, database, or server-side system. | API、worker、定时任务、服务、数据库或服务端系统。 |
| `cli` | Command-line tools and developer utilities. | 命令行工具和开发者工具。 |
| `data` | Scripts, notebooks, ETL, analytics, or machine learning repositories. | 脚本、Notebook、ETL、分析或机器学习仓库。 |
| `infra` | Terraform, Kubernetes, CI/CD, deployment, or operations repositories. | Terraform、Kubernetes、CI/CD、部署或运维仓库。 |
| `light` | Small scripts, examples, docs-only repos, or config repos. | 小型脚本、示例、纯文档或配置仓库。 |

The profile is not cosmetic. It controls how strict the output and validator should be. For example, an SDK should usually include public export rules and interface documentation, while a tiny script repository should not invent formal design documents just to satisfy a heavy template.

profile 不是装饰性标签，它会决定输出内容和校验规则的严格程度。例如 SDK 通常需要公开导出和接口文档规则，而小型脚本仓库不应该为了满足重型模板而编造正式设计文档。

## Output Scope / 输出内容范围

A generated `AGENTS.md` normally covers:

生成的 `AGENTS.md` 通常会覆盖：

- Project purpose and audience.
- Work boundaries and read-only references.
- Per-task development order.
- Requirements and specification entrypoints.
- Design document rules when the repo uses or needs durable design artifacts.
- Interface or API document rules when the repo exposes public contracts.
- Directory and module responsibilities.
- Coding conventions for the detected stack.
- Import/export or equivalent public surface rules.
- External dependency, SDK, service, platform, or infrastructure boundaries.
- Documentation and comment rules with examples.
- Testing, build, lint, format, verification, and environment rules.
- Git hygiene, delivery expectations, and final reporting rules.

- 项目定位和目标读者。
- 工作边界和只读参考对象。
- 每轮开发顺序。
- 需求和规格入口。
- 当仓库使用或需要长期设计资料时，补充设计文档规范。
- 当仓库暴露公开合同时，补充接口或 API 文档规范。
- 目录和模块职责。
- 与当前技术栈匹配的开发规范。
- import/export 或等价公开表面规则。
- 外部依赖、SDK、服务、平台或基础设施边界。
- 带示例的文档和注释规范。
- 测试、构建、lint、format、验证和环境规则。
- Git 规范、交付要求和最终汇报规则。

The output must stay project-wide. It should not say that one release, migration, branch, ticket, sprint, or temporary workaround is the permanent rule for the whole repository.

输出必须保持项目级长期规范视角，不应把某个版本、迁移、分支、ticket、迭代或临时方案写成整个仓库的永久规则。

## Validation Script / 校验脚本

The bundled validator is a zero-dependency Python smoke test. It checks whether an instruction file contains the expected structural guidance for a selected profile.

内置校验脚本是零依赖 Python 烟测工具，用于检查规范文件是否包含某个 profile 所需的核心结构。

Basic usage:

基础用法：

```bash
python3 scripts/validate_agents_md.py ./AGENTS.md --profile full
```

Common examples:

常见示例：

```bash
# SDK/library repository
# SDK 或库项目
python3 scripts/validate_agents_md.py ./AGENTS.md --profile sdk --forbid-transient

# Frontend, mobile, desktop, or full-stack application
# 前端、移动端、桌面端或全栈应用
python3 scripts/validate_agents_md.py ./AGENTS.md --profile app --forbid-transient

# Small script or docs-only repository
# 小型脚本或纯文档仓库
python3 scripts/validate_agents_md.py ./AGENTS.md --profile light

# Require project-specific text
# 要求必须出现某些项目特定内容
python3 scripts/validate_agents_md.py ./AGENTS.md --profile full --require "make test"

# Forbid task-specific wording
# 禁止某些一次性需求措辞
python3 scripts/validate_agents_md.py ./AGENTS.md --profile full --forbid "release-name" --forbid-transient
```

Useful flags:

常用参数：

| Flag | English | 中文 |
| --- | --- | --- |
| `--profile` | Selects validation profile. | 选择校验档位。 |
| `--require` | Requires a literal substring. | 要求出现某个字面字符串。 |
| `--require-regex` | Requires a regular expression match. | 要求匹配某个正则。 |
| `--forbid` | Fails if a literal substring appears. | 如果出现某个字面字符串则失败。 |
| `--forbid-regex` | Fails if a regular expression matches. | 如果匹配某个正则则失败。 |
| `--forbid-transient` | Fails on common ticket, sprint, branch, version, or temporary-plan wording. | 检测常见 ticket、迭代、分支、版本或临时方案措辞。 |
| `--allow-missing` | Allows a named check to be missing when intentionally smaller. | 对有意轻量化的仓库允许缺少某个检查项。 |
| `--alias` | Adds language-specific aliases for check names. | 为检查项补充其他语言的关键词。 |
| `--min-units` | Overrides minimum content size. | 覆盖最小内容长度要求。 |
| `--list-checks` | Lists all available check names. | 列出所有可用检查项。 |

The validator is intentionally not a semantic reviewer. Passing validation means the file has the expected structure and obvious forbidden phrases were not found. It does not prove that every rule is correct for the repository.

校验脚本不是语义审查器。通过校验只说明文件具备预期结构，并且没有发现明显禁用措辞；它不能证明每一条规则都完全符合项目事实。

## Good Use Cases / 适用场景

Use this skill when:

适合使用这个 skill 的情况：

- A repository has no `AGENTS.md`, but future agents need clear project rules.
- An existing `AGENTS.md` is too shallow, outdated, or mixed with one-time task notes.
- A project needs consistent development order, documentation expectations, and verification rules.
- A monorepo needs root-level guidance plus package-specific boundaries.
- A team wants examples that show what good and bad documentation or comments look like.

- 仓库还没有 `AGENTS.md`，但后续 Agent 需要清晰项目规范。
- 现有 `AGENTS.md` 太浅、过期，或混入了一次性需求记录。
- 项目需要统一开发顺序、文档要求和验证规则。
- monorepo 需要根级说明和各 package 的职责边界。
- 团队希望规范里有合格/不合格文档或注释示例。

## Limitations / 限制

- The skill cannot know repository facts that are not in the repository or provided by the user.
- The validator is structural, not semantic.
- If a repository has unusual internal conventions, the user should provide them explicitly.
- For non-Chinese and non-English output, use `--alias` in the validator when section names are written in another language.
- Installation depends on the Codex skill installer available in your local environment.

- 这个 skill 无法知道仓库里没有、用户也没有提供的项目事实。
- 校验脚本是结构检查，不是语义审查。
- 如果仓库有特殊内部规范，用户应明确提供。
- 如果输出不是中文或英文，而章节标题使用其他语言，校验时可通过 `--alias` 补充关键词。
- 安装方式取决于本地 Codex 环境可用的 skill 安装工具。
